1. CLAUDE.md 配置最佳实践

CLAUDE.md 是 Claude Code 的核心配置文件,它会在每次对话开始时自动加载到上下文中。用好这个文件,相当于给 Claude 配了一个项目说明书,能大幅提升代码生成的准确性和一致性。

什么是 CLAUDE.md?

CLAUDE.md 是一个特殊文件,Claude Code 启动时会自动读取并加入上下文。它的作用类似于项目的"使用说明书",可以包含:

  • • 常用命令和脚本 - 项目特有的 npm 脚本、构建命令、测试命令
  • • 核心文件与函数 - 关键模块的位置和功能说明
  • • 代码风格指南 - 团队约定的命名规范、文件组织方式
  • • 测试说明 - 如何运行测试、测试覆盖率要求
  • • 项目特殊约定 - 数据库连接方式、API 调用规范等
  • • 开发环境配置 - Node.js 版本、依赖安装注意事项

CLAUDE.md 的放置位置

CLAUDE.md 支持多层级放置,优先级从高到低:

1. 项目根目录

your-project/
├── CLAUDE.md # 团队共享,提交到 Git
├── CLAUDE.local.md # 个人配置,加入 .gitignore
└── src/

CLAUDE.md: 适合团队共享的配置,如代码规范、项目结构说明
CLAUDE.local.md: 个人偏好配置,不提交到 Git

2. 父目录(Monorepo)

monorepo/
├── CLAUDE.md # 整个 Monorepo 的通用配置
├── apps/
│ └── web/
│ └── CLAUDE.md # web 应用的专属配置
└── packages/
└── ui/
└── CLAUDE.md # UI 包的专属配置

Monorepo 场景下,可以在根目录放通用规范,在子项目放具体配置

3. 用户主目录

~/.claude/
└── CLAUDE.md # 全局个人配置

适合放通用的个人偏好,如代码风格、常用快捷命令,对所有项目生效

CLAUDE.md 内容示例

# 项目概述
本项目是一个基于 Next.js 14 的企业级 SaaS 应用。

## 技术栈
- Next.js 14 (App Router)
- TypeScript 5.3
- Tailwind CSS 3.4
- Prisma ORM
- PostgreSQL 16

## 代码风格
- 使用 ESLint + Prettier 进行代码格式化
- 组件文件使用 PascalCase 命名,如 `UserProfile.tsx`
- 工具函数使用 camelCase 命名
- 优先使用函数式组件和 Hooks
- 每个组件必须有对应的单元测试

## 常用命令
```bash
npm run dev # 启动开发服务器
npm run build # 生产环境构建
npm run test # 运行所有测试
npm run test:watch # 监听模式运行测试
npm run lint # 运行 ESLint
npm run format # 格式化所有代码
```

## 核心目录结构
```
src/
├── app/ # Next.js App Router 页面
├── components/ # 可复用组件
├── lib/ # 工具函数和配置
├── hooks/ # 自定义 Hooks
├── types/ # TypeScript 类型定义
└── prisma/ # 数据库 Schema 和迁移
```

## 重要约定
1. 所有 API 路由必须有错误处理和日志记录
2. 数据库查询必须使用 Prisma,禁止原始 SQL
3. 敏感信息(API Key)存放在 .env.local,不提交到 Git
4. 组件 props 必须定义 TypeScript 接口
5. 提交前必须通过 `npm run lint` 和 `npm run test`

## 测试要求
- 单元测试使用 Jest + React Testing Library
- 测试文件命名为 `ComponentName.test.tsx`
- 目标测试覆盖率: 80%以上

## 注意事项
- Node.js 版本必须 >= 18.17
- 首次运行需要执行 `npx prisma generate` 生成 Prisma Client
- 开发环境使用 SQLite,生产环境使用 PostgreSQL

💡 快速生成 CLAUDE.md

Claude Code 提供了 /init 命令,可以自动生成基础的 CLAUDE.md 文件:

# 在项目根目录运行
claude

# 然后输入命令
/init

Claude 会分析你的项目结构,自动生成包含技术栈、目录结构、package.json 脚本等信息的 CLAUDE.md 文件。你只需要在此基础上补充团队规范和特殊约定即可。

CLAUDE.md 编写技巧

✅ 应该包含的内容

  • • 项目特定信息: Claude 无法自动推断的约定和规范
  • • 常见错误警示: 容易踩的坑和注意事项
  • • 快捷命令: 提高开发效率的脚本和别名
  • • 架构模式: 项目采用的设计模式和最佳实践
  • • 环境依赖: 特定版本要求和配置说明

❌ 不应该包含的内容

  • • 通用知识: React、TypeScript 等基础概念,Claude 已经知道
  • • 过时信息: 不再维护的功能或已弃用的做法
  • • 冗余描述: package.json 已有的依赖列表无需重复
  • • 敏感信息: API Key、密码等,应该使用 .env 文件

2. 权限管理与安全配置

Claude Code 默认采用保守的权限策略:读取文件自动允许,但写入、执行 bash 命令需要手动确认。虽然这保证了安全性,但频繁确认会影响效率。合理配置权限规则,能在安全和效率之间找到平衡。

权限系统工作原理

三种权限规则类型

✓
allow (允许规则)

允许 Claude Code 自动使用指定工具,无需手动确认。适合安全且高频的操作,如读取项目文件、运行测试命令等。

?
ask (询问规则)

使用工具前询问用户确认,优先级高于 allow。适合需要谨慎处理的操作,如修改配置文件、执行数据库迁移等。

✕
deny (拒绝规则)

完全禁止 Claude Code 使用指定工具,优先级最高。适合保护敏感文件,如 .env、credentials.json、私钥文件等。

配置文件位置

权限配置保存在 .claude/settings.json 文件中:

# 项目级配置
your-project/.claude/settings.json

# 用户级全局配置
~/.claude/settings.json

# Windows 用户
C:\Users\用户名\.claude\settings.json

实用权限配置示例

1. 保护敏感文件

{
"permissions": {
"deny": [
"Read(./.env)",
"Read(./.env.*)",
"Read(./secrets/**)",
"Read(./config/credentials.json)",
"Read(~/.ssh/**)",
"Write(./.env)",
"Write(./package.json)" // 防止误修改依赖
]
}
}

2. 自动批准常用命令

{
"permissions": {
"allow": [
"Bash(npm run test)",
"Bash(npm run lint)",
"Bash(npm run build)",
"Bash(git status)",
"Bash(git diff)",
"Read(./src/**)",
"Read(./tests/**)"
]
}
}

3. 完整配置示例

{
"permissions": {
"deny": [
"Read(./.env*)",
"Read(./secrets/**)",
"Write(./.env*)",
"Bash(rm -rf *)",
"Bash(sudo *)"
],
"allow": [
"Read(./src/**)",
"Read(./tests/**)",
"Write(./src/**)",
"Write(./tests/**)",
"Bash(npm run test)",
"Bash(npm run lint)",
"Bash(git status)",
"Bash(git diff)"
],
"ask": [
"Write(./package.json)",
"Bash(npm install *)",
"Bash(git commit *)",
"Bash(npx prisma migrate *)"
]
}
}

⚠️ 跳过权限确认的风险

使用 claude --dangerously-skip-permissions 可以跳过所有权限确认,实现完全自动化。但这存在严重的安全风险:

  • • Claude 可能误删重要文件
  • • 可能执行危险的 bash 命令(如 rm -rf)
  • • 敏感文件可能被修改或泄露

建议: 仅在完全信任的私有项目中使用,并配合 Git 版本控制随时回滚。生产环境和公司项目绝不使用此选项。

3. 上下文管理与优化

虽然 Claude Code 拥有 200k tokens 的超大上下文窗口,但上下文累积过多仍然会影响响应速度和准确性。掌握上下文管理技巧,是高效使用 Claude Code 的关键。

为什么需要管理上下文?

  • • 响应变慢: 上下文越长,Claude 处理时间越久
  • • 精度下降: 无关历史信息可能干扰当前任务的判断
  • • 成本增加: 更多 tokens 消耗意味着更快达到使用限额
  • • 混淆风险: 旧任务的代码可能与新任务冲突

核心上下文管理命令

/clear - 清空上下文

删除所有历史对话和代码,但保留 CLAUDE.md 的内容。适合开始全新任务时使用。

# 使用场景
/clear # 清空历史,开始新任务

# 最佳实践: 每次切换任务时执行
# 例如: 写完一个功能 → /clear → 开始下一个功能

/compact - 压缩上下文

保留对话摘要,删除详细内容,大幅减少 token 占用。适合想保留任务历史但不需要完整细节的情况。

# 使用场景
/compact # 压缩历史对话,保留摘要

# 对比:
# /clear: 完全删除历史,轻装上阵
# /compact: 保留任务脉络,减少 token

快捷键: Escape - 中断操作

在 Claude 执行任务时按 Esc 可立即中断,保留上下文并允许你重新指导方向。

# 使用场景
# 1. Claude 理解错误,正在做无关的事 → 按 Esc 中断
# 2. 想补充更多指令 → 按 Esc,然后输入新指令
# 3. 发现更好的方案 → 按 Esc,调整方向

# 双击 Esc: 回退到历史某个节点,重新选择路径

上下文管理最佳实践

1. 养成定期清理习惯

  • • 每次切换任务前: 执行 /clear
  • • 长时间开发后: 每 1-2 小时执行 /compact
  • • 发现响应变慢时: 立即 /clear 重新开始

2. 拆分大任务为小任务

与其让 Claude 一次性完成复杂功能,不如拆分成多个小步骤:

❌ 不好的做法:
"帮我实现用户认证、权限管理、日志记录和邮件通知功能"

✅ 推荐做法:
1. "先实现用户注册和登录" → 完成后 /clear
2. "添加基于角色的权限控制" → 完成后 /clear
3. "集成日志记录中间件" → 完成后 /clear
4. "实现邮件通知服务" → 完成后 /clear

3. 使用明确的任务描述

清晰的任务描述能让 Claude 更快理解意图,减少无效对话:

❌ 模糊描述:
"优化一下这个组件"

✅ 明确描述:
"优化 UserProfile 组件:
1. 提取重复的样式到 Tailwind 类
2. 使用 useMemo 缓存计算结果
3. 懒加载头像图片
4. 添加加载状态和错误处理"

4. 高效工作流技巧

Claude Code 的强大在于其自主性,但如何引导它高效工作,是一门艺术。以下是经过实战验证的工作流技巧。

核心原则: 先计划,再执行

为什么要先让 Claude 制定计划?

让 Claude 直接动手写代码,容易出现方向偏差、遗漏边缘情况、代码结构混乱等问题。要求它先输出计划,你可以:

  • • 提前发现理解偏差,及时纠正
  • • 评估技术方案的合理性
  • • 补充遗漏的需求和边缘情况
  • • 了解实现步骤,做到心中有数

推荐工作流

1
描述任务,要求制定计划
"我需要实现一个用户评论功能,包括发表评论、回复评论、点赞和举报。
请先制定详细的实现计划,包括:
1. 数据库 Schema 设计
2. API 端点设计
3. 前端组件结构
4. 需要考虑的边缘情况

在我确认计划后再开始编码。"
2
审查计划,提出调整
"计划整体不错,但有几点需要调整:
1. Schema 中需要加上 isDeleted 字段支持软删除
2. API 需要添加分页和排序功能
3. 前端组件要支持无限滚动加载

请更新计划,然后开始实现。"
3
Claude 自主执行,使用 TODO 跟踪进度

Claude Code 内置了 TODO 列表功能,会自动创建和更新任务清单:

# Claude 自动生成的 TODO 列表
[x] 创建评论数据库 Schema
[x] 实现评论 API - POST /api/comments
[>] 实现前端评论表单组件 # 正在进行
[ ] 添加单元测试
[ ] 集成到用户界面

按 Esc 可以随时中断,查看进度或调整方向

4
执行测试验证
"请运行以下测试确保功能正常:
1. npm run test
2. npm run lint
3. 手动测试: 创建评论、回复、点赞、举报

如果测试失败,请自动调试并修复。"

测试驱动开发(TDD)最佳实践

Claude Code 非常适合 TDD 工作流。让它先写测试,再实现功能,能大幅提升代码质量:

"使用 TDD 方式实现用户登录功能:
1. 先编写测试用例:
- 正确的用户名和密码应该登录成功
- 错误的密码应该返回 401
- 不存在的用户应该返回 404
- 缺少必填字段应该返回 400
2. 运行测试,确保失败(Red)
3. 实现最小化代码使测试通过(Green)
4. 重构代码,保持测试通过(Refactor)

每完成一个测试用例,都要运行 npm run test 验证。"

Git 工作流集成

Claude Code 可以自动执行 Git 操作,但建议保持人工审查:

# 推荐做法: 每完成一个功能模块就提交
"完成评论功能后,请:
1. 运行 git status 检查修改的文件
2. 生成一条清晰的 commit message
3. 不要自动提交,把 git add 和 commit 命令告诉我,我手动执行"

# 或者让 Claude 提交,但要求详细说明
"提交代码前,请先告诉我:
1. 修改了哪些文件
2. 主要改动内容
3. 建议的 commit message
我确认后你再执行 git commit"

5. 高级功能与深度思考

Claude Code 内置了一些强大的高级功能,合理使用能解决更复杂的编程挑战。

Extended Thinking(深度思考模式)

什么是深度思考?

当任务特别复杂时,Claude 可以进入"深度思考"模式,分配更多计算资源进行推理。这类似于人类遇到难题时会花更多时间思考。

如何激活深度思考?

在提示词中使用"think"相关词汇,Claude 会自动调整思考深度:

思考级别
  • • think - 基础深度思考
  • • think hard - 中等深度
  • • think harder - 高深度
  • • ultrathink - 最高深度
适用场景
  • • 复杂算法设计
  • • 架构重构方案
  • • 性能优化策略
  • • 安全漏洞分析
# 使用示例
"这是一个复杂的分布式锁实现问题,涉及 Redis、Lua 脚本和并发控制。
请 think harder 后给出最优方案,需要考虑:
1. 死锁预防
2. 锁超时处理
3. 主从切换时的一致性
4. 性能优化"

# Claude 会进入深度思考模式,输出更详细的分析和方案

Checkpoint(检查点功能)

什么是检查点?

检查点允许你在关键节点保存项目状态,如果后续出现问题,可以回滚到检查点重新开始,而不是从头再来。

使用场景

  • • 尝试性重构: 不确定方案是否可行,先打个检查点再试
  • • 多方案对比: 实现方案 A,不满意就回滚,尝试方案 B
  • • 风险操作: 大规模删除或修改前,先保存检查点
# 使用示例
"在开始重构数据库层之前,请创建一个检查点。
如果重构后测试失败,我们可以回滚重新思考。"

# Claude 会记住当前状态
# 如果需要回滚:
"回滚到重构前的检查点,我们换一种方案"

Memory Tools(记忆工具)

跨会话记忆

Claude Code 可以在不同会话之间记住项目偏好、历史决策和解决方案,避免重复回答同样的问题。

记忆的内容

  • • 你的编码偏好(如函数式 vs 面向对象)
  • • 项目特定的技术决策
  • • 过去遇到的问题和解决方法
  • • 你的工作流习惯
# 第一次会话
你: "我更喜欢使用函数式编程风格,优先使用 map/filter/reduce"
Claude: "好的,我会记住这个偏好,后续生成代码时会遵循函数式风格"

# 下次会话,Claude 会自动应用这个偏好
# 无需再次说明

6. 自定义命令与自动化

Claude Code 支持自定义命令,你可以把常用的操作流程封装成命令,一键触发,大幅提升效率。

自定义命令目录结构

# 用户级命令(所有项目通用)
~/.claude/commands/
├── review.md # /user:review
├── refactor.md # /user:refactor
└── optimize.md # /user:optimize

# 项目级命令(仅当前项目)
your-project/.claude/commands/
├── test.md # /project:test
├── deploy.md # /project:deploy
└── migrate.md # /project:migrate

自定义命令示例

示例 1: 代码审查命令

文件: ~/.claude/commands/review.md

请对当前更改的代码进行全面审查,检查以下方面:

1. **代码质量**
- 是否遵循项目代码风格
- 变量命名是否清晰
- 函数是否符合单一职责原则

2. **潜在 Bug**
- 边缘情况处理
- 错误处理是否完善
- 空值/undefined 检查

3. **性能优化**
- 是否存在不必要的重复计算
- 数据库查询是否优化
- 是否可以使用缓存

4. **安全性**
- 输入验证
- SQL 注入风险

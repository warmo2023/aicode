1. 基础交互：会话管理与命令行操作

Claude Code 的核心交互方式是命令行。掌握这些基础命令，是高效使用的第一步。

1.1 启动和退出

# 进入项目目录并启动 Claude Code
cd /path/to/your/project
claude

# 执行一次性任务（无需进入交互模式）
claude "帮我分析这个项目的代码结构"

# 继续上次会话
claude -c

# 恢复历史会话
claude -r

# 退出
exit # 或按 Ctrl+C

💡 实用技巧

使用 claude "task" 适合快速任务，比如"生成测试数据"或"检查代码规范"。对于需要多轮对话的复杂任务，还是进入交互模式更高效。

1.2 会话控制命令

在交互模式下，这些命令帮你管理对话状态：

/clear # 清除当前对话上下文（但不清除代码修改）
/status # 查看会话详细信息（token 使用、配置、历史）
/help # 查看所有可用命令
exit # 退出当前会话

其中 /status 是 v2.0 新增的强大功能，它提供三个标签页：

  • 会话信息：当前对话的 token 消耗、消息数量、模型版本
  • 配置详情：加载的 CLAUDE.md、已启用的插件、MCP 服务器连接
  • 使用统计：累计 token 使用、成本估算（如果配置了 API）

⚠️ 注意事项

/clear 只清除对话历史，不会回滚代码修改。如果想撤销代码改动，需要使用下面要讲的 Rewind 功能。

2. Rewind 功能：随时回滚代码和对话

这是 Claude Code v2.0 最实用的新特性之一。开发过程中，AI 可能会改错方向，或者你想测试不同方案——Rewind 让你可以瞬间回到之前的任何时间点。

2.1 如何激活 Rewind

  • 快捷键：连续按两次 ESC
  • 命令：输入 /rewind

激活后，Claude Code 会显示一个时间线界面，列出所有的"检查点"（通常是每次 Claude 执行工具调用时自动创建）。你可以选择任意一个检查点，系统会同时回滚：

  • 对话上下文（之后的消息全部清除）
  • 代码状态（通过 Claude 工具修改的文件会恢复）

2.2 哪些修改可以回滚？

✅ 可以回滚的操作：

  • 通过 Claude 的 Edit 工具修改的文件
  • 通过 Claude 的 Write 工具创建的文件
  • 通过 Claude 删除的文件

❌ 无法回滚的操作：

  • 通过 Bash 命令修改的文件（如 rm file.txt）
  • 通过 Bash 命令移动或复制的文件（如 mv old.txt new.txt）
  • 手动编辑器中做的修改

2.3 实战场景

假设你让 Claude 重构一个函数，它改了10个文件，但你发现思路不对。这时：

  1. 按两次 ESC 或输入 /rewind
  2. 选择"开始重构"之前的检查点
  3. 所有10个文件瞬间恢复原样，对话也回到了那个时间点
  4. 重新给 Claude 更明确的指令，走另一条实现路线

这比手动 git reset 或逐个撤销文件快得多，而且不会影响 Git 历史。

3. 扩展思维模式：让 Claude 深度思考

Claude Code 有一个"隐藏能力"：扩展思维（Extended Thinking）。默认情况下，Claude 会快速响应，但对于复杂的架构决策或调试问题，你可以要求它花更长时间深度分析。

3.1 激活方式

在提示词中加入特定关键词：

  • think - 基础深度思考
  • think hard - 中等强度思考
  • think harder - 高强度思考
  • ultrathink - 最强思考模式

3.2 示例对话

# 基础询问（快速响应）
> 如何在这个项目中添加 OAuth2 认证？

# 启用扩展思维（Claude 会给出更深入的架构分析）
> 我需要为这个 API 实现 OAuth2 认证系统。Think deeply about the best approach, considering our current tech stack, security requirements, and scalability.

# 针对特定问题深度思考
> think hard about the edge cases in this authentication flow that could lead to security vulnerabilities

3.3 适用场景

  • 系统架构设计：选择技术栈、设计数据库结构、规划微服务拆分
  • 复杂 Bug 调试：偶发性问题、多线程竞争、内存泄漏
  • 性能优化：找出瓶颈点、评估不同方案的时间复杂度
  • 重构决策：是否重构、如何重构、影响范围评估

💡 使用建议

扩展思维会消耗更多 token 和时间，只在真正需要深度分析时使用。简单的代码补全或格式化，用默认模式就够了。如果你经常需要长时间会话，通过 GACCode 订阅可以获得更稳定的 token 配额，避免官方限制。

4. 自定义命令：打造专属工作流

自定义命令（Slash Commands）是 Claude Code 的"编程接口"。你可以把常用的任务封装成命令，一键触发，极大提升效率。

4.1 命令的三种作用域

作用域 路径 适用场景
个人命令 ~/.claude/commands/ 所有项目通用的任务
项目命令 .claude/commands/ 当前项目特有的流程
插件命令 插件的 commands/ 目录 第三方插件提供

4.2 创建你的第一个命令

示例：创建一个代码安全审查命令

# 创建个人命令目录
mkdir -p ~/.claude/commands

# 创建命令文件
cat > ~/.claude/commands/security-review.md << 'EOF'
---
description: 审查代码安全漏洞
allowed-tools: Read, Grep
---

# 安全审查任务

请审查当前项目的代码，重点关注：

1. SQL 注入风险
2. XSS 跨站脚本攻击
3. 未经验证的用户输入
4. 硬编码的密钥和凭证
5. 不安全的文件上传

给出具体的代码位置和修复建议。
EOF

现在在 Claude Code 中输入：

> /security-review

Claude 会自动按照你定义的流程执行安全审查。

4.3 命令参数

命令可以接收参数，有两种方式：

  • $ARGUMENTS - 捕获所有参数
  • $1, $2, $3 - 单个位置参数

示例：创建一个修复 Issue 的命令

cat > .claude/commands/fix-issue.md << 'EOF'
---
argument-hint: [issue-number]
description: 修复指定的 GitHub Issue
---

# 修复 Issue #$ARGUMENTS

1. 阅读 Issue 描述，理解问题
2. 定位相关代码
3. 实现修复方案
4. 添加测试用例
5. 准备 PR 描述
EOF

使用：

> /fix-issue 123

Claude 会自动将 123 替换到 $ARGUMENTS 位置。

4.4 Bash 模式：在命令中执行 Shell

命令文件中可以用 ! 前缀执行 bash 命令，结果会自动添加到上下文：

---
allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*)
description: 创建 Git 提交
---

## 当前状态

- Git 状态：!`git status`
- 待提交更改：!`git diff HEAD`
- 当前分支：!`git branch --show-current`

## 任务

根据上述更改，创建一个描述清晰的 Git 提交。

这样 Claude 在执行命令时，会先运行这些 bash 命令，获取实时的 Git 状态，再基于这些信息生成提交。

💡 高级技巧

把团队的代码规范、测试流程、部署步骤都封装成项目命令，新成员可以直接用 /deploy、/test 等命令，无需记忆复杂的操作流程。

5. CLAUDE.md 配置文件：项目上下文管理

CLAUDE.md 是一个特殊文件，放在项目根目录的 .claude/ 目录下。每次启动 Claude Code，它会自动加载这个文件的内容，作为"项目说明书"。

5.1 应该写什么？

一个完整的 CLAUDE.md 通常包含：

  • 项目架构：目录结构、关键模块说明
  • 代码风格：缩进、命名规范、注释要求
  • 常用命令：构建、测试、部署的 npm/yarn 命令
  • 核心文件和函数：重要的工具函数、配置文件位置
  • 开发规范：分支策略、提交信息格式、PR 流程
  • 调试技巧：常见问题的排查方法

5.2 示例：一个前端项目的 CLAUDE.md

# MyApp 项目指南

## 技术栈

- React 18 + TypeScript
- Vite 作为构建工具
- Tailwind CSS 用于样式
- React Query 管理服务端状态

## 目录结构

```
src/
components/ # 可复用组件
pages/ # 页面组件
hooks/ # 自定义 Hooks
utils/ # 工具函数
api/ # API 调用封装
```

## 代码规范

- 组件名使用 PascalCase
- 文件名与组件名一致
- 每个组件一个文件
- Props 类型必须定义
- 使用函数组件 + Hooks，禁止 Class 组件

## 常用命令

- `npm run dev` - 启动开发服务器
- `npm run build` - 生产构建
- `npm test` - 运行测试
- `npm run lint` - 代码检查

## 核心工具

- `src/utils/request.ts` - 封装的 fetch 函数，所有 API 调用都通过它
- `src/hooks/useAuth.ts` - 用户认证状态管理

## Git 规范

- 提交信息格式：`type(scope): description`
- 类型：feat, fix, docs, style, refactor, test, chore
- 示例：`feat(auth): add OAuth login support`

5.3 为什么要用 CLAUDE.md？

有了这个文件，Claude 在任何对话中都能：

  • 按照项目的代码风格生成代码
  • 知道去哪里找关键文件
  • 理解项目的架构和技术选型
  • 给出符合团队规范的建议

这相当于给 Claude 提供了一份"入职文档"，让它快速融入你的项目。

⚠️ 注意

CLAUDE.md 的内容会被计入每次对话的 token，所以不要写得过长。控制在 2000 词以内，只写最核心的信息。

6. Git 深度集成：自然语言操作版本控制

Claude Code 不只是写代码，它还能直接操作 Git——而且是用自然语言。

6.1 自然语言 Git 命令

你可以这样跟 Claude 说话：

> 我改了哪些文件？
> 用一句话总结我的修改，然后提交
> 创建一个新分支叫 feature/user-profile
> 显示最近5次提交
> 帮我解决合并冲突

Claude 会自动转换为对应的 Git 命令并执行。

6.2 Git Worktrees：并行开发的神器

如果你需要同时开发多个功能（比如一个紧急修复 + 一个新特性），频繁切换分支会很烦。Claude Code 支持 Git Worktrees，让你在不同目录下并行工作。

# 为新功能创建 worktree
git worktree add ../project-feature-a -b feature-a

# 为 bug 修复创建另一个 worktree
git worktree add ../project-bugfix -b bugfix-123

# 在第一个 worktree 中启动 Claude
cd ../project-feature-a
claude

# 在第二个 worktree 中启动另一个 Claude 会话
cd ../project-bugfix
claude

两个 Claude 会话完全独立，互不干扰。功能开发完成后，清理 worktree：

# 查看所有 worktree
git worktree list

# 删除不需要的 worktree
git worktree remove ../project-feature-a

7. MCP 协议：连接外部工具和数据源

MCP（Model Context Protocol）是 Anthropic 设计的一种标准协议，让 Claude 可以连接外部工具、数据库、API 等。通过 MCP，Claude Code 能做的事情远超代码编辑。

7.1 支持的传输方式

  • HTTP：通过 HTTP API 连接
  • SSE（Server-Sent Events）：实时数据流
  • stdio：标准输入输出（本地工具）

7.2 添加 MCP 服务器

从 Claude Desktop 导入（macOS/WSL）：

claude mcp add-from-claude-desktop

手动添加 HTTP 服务器：

claude mcp add --transport http github https://api.github.com/mcp

添加本地工具（stdio）：

claude mcp add --transport stdio jira -- /usr/local/bin/jira-mcp-tool

7.3 使用 MCP 提示

MCP 服务器提供的功能会自动转换为 slash 命令，格式为：

/mcp__<server-name>__<prompt-name> [arguments]

示例：

# 列出 GitHub PR
> /mcp__github__list_prs

# 审查特定 PR
> /mcp__github__pr_review 456

# 创建 Jira Issue
> /mcp__jira__create_issue "登录页面崩溃" high

7.4 权限配置

MCP 工具需要权限配置：

  • mcp__github - 允许该服务器的所有工具
  • mcp__github__get_issue - 只允许特定工具
  • 不支持通配符（mcp__github__* 无效）

💡 实用场景

连接 Notion MCP 服务器，让 Claude 直接读取你的项目文档；连接数据库 MCP，让它分析数据库结构和优化查询。MCP 的潜力远超想象。

12. Hooks 系统：自定义工具调用逻辑

Hooks 允许你在工具调用前后插入自定义逻辑，比如日志记录、命令验证、数据转换。

示例：记录所有 Bash 命令

# 在 ~/.claude/settings.json 中配置
{
"hooks": {
"PreToolUse": [
{
"matcher": "Bash",
"hooks": [
{
"type": "command",
"command": "jq -r '\"\\(.tool_input.command)\"' >> ~/.claude/bash-log.txt"
}
]
}
]
}
}

这样每次 Claude 执行 bash 命令，都会先记录到日志文件。

为什么 Claude Code 会"犯蠢"？

1. 上下文混乱或溢出：对话历史过长，导致早期的关键信息被遗忘。

2. CLAUDE.md 文件缺失或不规范：没有提供项目规范和约定，AI 只能依靠通用知识推测。

3. 需求描述模糊：提示词过于简略或含糊，AI 无法准确理解意图。

4. 缺乏业务上下文：AI 不了解项目的业务逻辑和架构设计。

5. 工具使用不当：未能充分利用 Claude Code 的内置功能和最佳实践。

技巧 1：建立完善的 CLAUDE.md 文件

CLAUDE.md 文件是 Claude Code 最重要的配置机制。这个文件会在每次对话启动时自动加载到上下文中，为 AI 提供持久化的项目知识。

CLAUDE.md 文件应包含的核心内容

1. 常用命令

# 常用命令
## 开发服务器
npm run dev

## 运行测试
npm test -- --coverage

## 构建生产版本
npm run build

## 代码检查
npm run lint

2. 核心文件位置

# 核心文件
- src/utils/api.ts - API 请求封装
- src/components/Layout.tsx - 全局布局组件
- src/hooks/useAuth.ts - 认证管理 Hook
- config/database.ts - 数据库配置

3. 代码风格规范

# 代码风格
- 使用 TypeScript 严格模式
- 组件使用函数式组件 + Hooks
- CSS 使用 Tailwind CSS
- 命名约定：
  - 组件：PascalCase.tsx
  - 工具函数：camelCase.ts
  - 常量：UPPER_SNAKE_CASE

4. 测试要求

# 测试规范
- 单元测试使用 Jest
- E2E 测试使用 Playwright
- 目标覆盖率：80% 以上
- 每个 API 路由必须有集成测试

实战建议

CLAUDE.md 文件支持多层级配置。除了在项目根目录创建全局配置，还可以在子目录（如 frontend/、backend/）创建局部配置文件，提供模块级别的上下文。Claude Code 会递归读取并合并这些文件。

技巧 2：使用 TODO.md 拆解任务

任务拆解是避免 AI "走偏"的关键。通过创建 TODO.md 文件，将复杂需求拆解为清晰的子任务列表，可以帮助 Claude Code 保持专注，按步骤执行。

如何使用 TODO.md

步骤 1：让 Claude 创建 TODO.md

"请根据当前需求创建一个 TODO.md 文件，将任务拆解为具体的步骤"

步骤 2：审核任务列表

检查拆解的任务是否合理、完整，是否遗漏了关键步骤。

步骤 3：逐步执行

要求 Claude 按照 TODO.md 中的顺序执行任务，完成一个勾选一个。

步骤 4：自动更新

"在执行过程中自动更新 TODO.md，标记已完成的任务"

TODO.md 示例

# 用户认证功能开发

## 任务列表
- [x] 1. 创建用户数据模型（User Model）
- [x] 2. 实现注册 API 端点
- [ ] 3. 实现登录 API 端点
- [ ] 4. 添加 JWT Token 认证中间件
- [ ] 5. 创建前端注册表单组件
- [ ] 6. 创建前端登录表单组件
- [ ] 7. 编写单元测试
- [ ] 8. 编写集成测试

## 当前进度
正在执行：任务 3（实现登录 API 端点）

## 注意事项
- 密码必须使用 bcrypt 加密
- Token 有效期设置为 7 天
- 前端表单需要实时验证

技巧 3：定期清理上下文

上下文溢出是 Claude Code 性能下降的主要原因。当对话历史过长时，早期的重要信息会被遗忘，响应速度也会变慢。定期使用 /clear 命令清理上下文至关重要。

何时使用 /clear 命令

完成独立任务后：每个独立功能开发完成后，清理上下文避免干扰下一个任务。
切换项目或模块：从前端切换到后端开发时，清理旧的上下文。
AI 响应变慢或出错：如果发现 Claude 开始遗忘信息或频繁出错，立即清理上下文。
对话轮次超过 20 轮：作为经验法则，超过 20 轮对话后考虑清理。

清理前的准备工作

在使用 /clear 之前，确保：

  • 重要的项目规范和约定已写入 CLAUDE.md 文件
  • 当前任务已完成或记录到 TODO.md 中
  • 关键决策和代码位置已文档化

这样清理后重新开始对话时，CLAUDE.md 会自动加载，无需重复说明项目信息。

技巧 4：精简 CLAUDE.md 文件

CLAUDE.md 文件会在每次对话时加载到上下文中。如果文件过于庞大，反而会消耗大量上下文空间，导致其他重要信息无法加载。因此需要定期审查和精简这个文件。

CLAUDE.md 文件臃肿的症状

  • 文件超过 500 行或 10KB
  • 包含大量过时的指令和规范
  • 有重复或冲突的规则说明
  • 包含具体的代码示例（应该移到文档）
  • Claude 频繁忽略文件中的指令

精简 CLAUDE.md 的最佳实践

1. 只保留关键规范

删除显而易见的规范（如"代码要有注释"），只保留项目特有的约定。

2. 使用分层配置

将全局规范放在根目录的 CLAUDE.md，模块特定规范放在子目录的 CLAUDE.md。

3. 定期归档

项目演进过程中，将过时的规范移动到 ARCHIVE.md 文件中保存，不加载到上下文。

4. 使用简洁语言

❌ 冗长：
"在编写组件时，请务必确保使用 TypeScript 严格类型检查，并且为每个 props 定义明确的类型接口..."

✅ 简洁：
"组件：使用 TS 严格模式，props 必须定义类型接口"

技巧 5：使用 # 键快速添加指令

Claude Code 提供了一个非常实用的功能：按下 # 键可以快速将当前指令添加到 CLAUDE.md 文件中。这个功能特别适合在编码过程中随时记录重要的项目约定。

# 键功能的使用场景

场景 1：发现新的项目约定

在代码审查中发现了新的命名规范，立即按 # 键记录：

"API 路由文件统一使用 route.ts 命名" + 按 # 键

场景 2：记录错误教训

发现 Claude 犯了某个错误，立即添加约束：

"数据库操作必须使用事务包裹，避免数据不一致" + 按 # 键

场景 3：强调关键依赖

项目中使用了特定的库或工具，需要 Claude 记住：

"日期处理统一使用 day.js，禁止使用 moment.js" + 按 # 键

注意事项

使用 # 键添加的指令会追加到 CLAUDE.md 文件末尾。定期审查这些新增内容，整理到合适的分类下，避免文件结构混乱。建议每周至少审查一次 CLAUDE.md，保持其清晰有序。

技巧 6：强化关键指令

AI 模型对某些关键词更敏感。在 CLAUDE.md 或提示词中，对于绝对不能违反的规则，使用强调性关键词可以显著提升遵循度。

有效的强调关键词

强制性指令

  • IMPORTANT：重要规则
  • YOU MUST：必须执行
  • NEVER：绝对禁止
  • ALWAYS：始终遵循

优先级标记

  • [CRITICAL]：关键约束
  • [REQUIRED]：必需项
  • [FORBIDDEN]：禁止行为
  • [MANDATORY]：强制要求

CLAUDE.md 强化示例

# 项目规范

## 代码风格
IMPORTANT: 所有代码必须使用 TypeScript 严格模式。

## 数据库操作
[CRITICAL] YOU MUST 在所有涉及多表操作的场景中使用数据库事务。
NEVER 直接使用原生 SQL，必须通过 Prisma ORM。

## 安全规范
[FORBIDDEN] 绝对禁止在代码中硬编码 API 密钥或敏感信息。
ALWAYS 使用环境变量管理配置。

## 测试要求
[REQUIRED] 每个 API 端点必须包含至少一个集成测试。
YOU MUST 在提交代码前运行完整的测试套件。

技巧 7：具体化需求描述

模糊的需求描述是 AI "理解错误"的主要原因。越具体的需求，AI 越能准确理解和执行。避免使用含糊的表述，提供明确的约束和预期结果。

模糊 vs 具体的需求对比

❌ 模糊需求：
"优化这个函数的性能"

✅ 具体需求：
"这个函数处理 10 万条数据时耗时超过 5 秒。请优化查询逻辑，添加索引，或使用批处理，将执行时间降低到 1 秒以内。"

❌ 模糊需求：
"添加用户认证功能"

✅ 具体需求：
"实现基于 JWT 的用户认证。需要包含：1) 注册接口（邮箱+密码），2) 登录接口（返回 JWT token），3) 认证中间件（验证 token 有效性），4) 密码使用 bcrypt 加密，5) token 有效期 7 天。"

❌ 模糊需求：
"修复这个 bug"

✅ 具体需求：
"在 Chrome 浏览器移动端视图下，用户点击'提交订单'按钮时页面无响应。控制台显示 'Cannot read property userId of undefined' 错误。请修复这个问题，确保在用户未登录时显示友好的提示信息。"

编写具体需求的 5W1H 原则

What（做什么）：明确要实现的功能或解决的问题。

Why（为什么）：说明业务背景和目的。

Where（在哪里）：指定文件位置或代码模块。

When（何时触发）：描述执行条件或时机。

Who（影响谁）：说明用户角色或使用场景。

How（如何验证）：提供验收标准或测试方法。

技巧 8：让 AI 先理解业务

在执行具体任务前，先让 Claude Code 理解项目的业务逻辑和架构设计。这是避免 AI "南辕北辙"的关键步骤。

业务理解的三个层次

层次 1：项目背景

在对话开始时，简要说明项目是什么、解决什么问题。

"这是一个在线教育平台，用户可以购买课程、观看视频、提交作业。主要用户群体是大学生和职场人士。"

层次 2：架构设计

说明技术栈、模块划分、数据流向。

"前端使用 Next.js，后端 API 使用 Express，数据库是 PostgreSQL。前后端通过 RESTful API 通信，认证使用 JWT。"

层次 3：当前任务的业务上下文

解释当前任务在整个业务流程中的位置和作用。

"现在需要实现'课程购买'功能。用户点击'购买'按钮后，调用支付接口，支付成功后在数据库中创建订单记录，并发送邮件通知用户。"

实战技巧

在开始新功能开发前，可以要求 Claude 先"复述"一遍业务逻辑和实现方案：

"在开始编码前，请用自己的话总结一下：1) 这个功能要解决什么问题，2) 涉及哪些模块和文件，3) 数据流是怎样的，4) 有哪些边界情况需要处理。"

这个步骤可以帮助你发现 AI 是否真正理解了需求，避免后续出现大量返工。

技巧 9：掌握错误恢复技巧

使用 Claude Code 过程中难免遇到错误。掌握快速恢复技巧可以大幅提升工作效率，避免因错误而中断开发流程。

常见错误类型及处理方法

错误 1：overloaded_error（服务器过载）

症状：提示服务器繁忙，请求被拒绝。

解决方法：

  • 等待 30 秒 - 1 分钟后重试
  • 使用 GACCode 提供的专业服务，避免官方 API 的高峰拥堵问题
  • 检查网络连接是否稳定

错误 2：request timeout（请求超时）

症状：请求发送后长时间无响应。

解决方法：

  • 按 Esc + Esc 回退到上一条消息
  • 简化请求，拆分为更小的任务
  • 检查 CLAUDE.md 是否过于庞大，占用了过多上下文

错误 3：tool_call_error（工具调用错误）

症状：Claude 尝试执行命令或访问文件时失败。

解决方法：

  • 检查文件路径是否正确
  • 确认当前目录权限
  • 按 Ctrl + C 强制退出，然后重启 Claude Code

错误 4：invalid_request_error（无效请求）

症状：请求格式不正确或参数无效。

解决方法：

  • 检查 API 密钥是否正确配置
  • 使用 GACCode 提供的可靠 API 服务，避免配置错误
  • 清理浏览器缓存或重启 CLI

使用稳定服务避免频繁错误

许多错误源于 API 服务的不稳定。通过 GACCode 订阅专业的 Claude Code 服务，可以有效规避官方平台的限流、封号等政策风险，确保开发流程的连续性和稳定性。

技巧 10：开启权限旁路模式

权限旁路模式（Bypassing Permissions）可以大幅提升工作效率。开启后，Claude Code 执行文件操作和命令时不再需要逐一授权，减少了频繁确认的中断。

如何开启权限旁路模式

方法 1：命令行参数

claude-code --dangerously-skip-permissions

方法 2：配置文件

在项目根目录的 .claude/config.json 中添加：

{
  "skipPermissions": true
}

安全警告

开启权限旁路模式后，Claude Code 可以执行任意命令和文件操作。务必遵循以下安全准则：

  • 只在熟悉的项目中使用
  • 使用版本控制（Git），随时可以回滚
  • 定期检查 Claude 执行的命令和文件修改
  • 避免在生产环境或敏感项目中使用
  • 备份重要数据

推荐使用场景

  • 原型开发：快速验证想法，不需要频繁确认
  • 重复性任务：批量修改文件或执行测试
  • 个人项目：已熟悉项目结构，风险可控
  • 学习实验：探索新技术，不涉及重要数据

技巧 12：使用 /resume 恢复对话

意外退出对话后，使用 /resume 命令可以恢复之前的会话。这个功能在长时间开发过程中非常实用，避免了重新建立上下文的麻烦。

/resume 命令的使用场景

场景 1：意外关闭终端

不小心关闭了 Claude Code 会话窗口，使用 /resume 立即恢复。

场景 2：切换任务后返回

临时处理其他事务，回来后使用 /resume 继续之前的开发任务。

场景 3：跨设备开发

在另一台电脑上登录后，使用 /resume 恢复最近的会话（需要云端同步支持）。

注意事项

  • /resume 只能恢复最近一次的会话
  • 如果执行了 /clear 命令，之前的对话无法恢复
  • 会话历史有时效性，建议在中断后尽快恢复
  • 重要的项目信息应写入 CLAUDE.md，而非依赖会话恢复
